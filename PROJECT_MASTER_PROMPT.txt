INDDIA ERP MASTER PROJECT PROMPT

Use this prompt when you want an AI assistant, developer, or implementation agent to understand the INDDIA ERP project deeply and continue work consistently.

PROJECT IDENTITY

Project name: INDDIA ERP
Project type: School ERP / School Management Platform
Primary purpose: A production-oriented multi-role ERP platform for schools, with a super-admin layer for platform-wide control and school-level dashboards for operations.
Current state: Active product build with broad frontend coverage, shared dashboard shell, Firebase-based auth and Firestore integration in progress, plus remnants of an older Supabase-oriented data shape in some service and type layers.

HIGH-LEVEL PRODUCT GOAL

Build and maintain a scalable school ERP platform that supports:
- Platform owner operations through a Super Admin console
- School administration through an Admin dashboard
- Faculty workflows through Staff/Teacher dashboards
- Student access for academic visibility
- Parent access for child monitoring
- Mobile-friendly usage and Capacitor readiness

The system should feel like a full operational platform, not a demo. It should support real workflows for attendance, fees, classes, timetable, staffing, billing, transport, analytics, notifications, subscription management, and audit visibility.

CURRENT ARCHITECTURE SUMMARY

The app is a single-page React application with shared layout shells and role-protected routes.

Core entry points:
- src/main.tsx mounts the app, router, and global language provider
- src/App.tsx contains routing, auth restore logic, protected/public routes, and role-aware navigation flows
- src/components/layout/DashboardLayout.tsx is the shared shell for main dashboard and super-admin dashboard
- src/components/layout/Navbar.tsx contains topbar controls, notifications, logout, and the global language icon
- src/store/authStore.ts manages user, role, school, session, and loading state via Zustand
- src/services/authService.ts handles login/session restoration and auth hydration
- src/services/adminService.ts is the broad service layer for ERP operations
- src/services/firebaseClient.ts initializes Firebase app/auth/firestore

AUTHENTICATION AND SESSION MODEL

The app uses Firebase Auth where configured, but it also includes a bootstrap fallback super admin.

Important auth concepts:
- Zustand auth store holds user, role, school, session, and loading
- App boot attempts restoreSession
- onAuthStateChanged hydrates or clears state
- If Firebase is absent or still being wired, bootstrap super admin fallback may allow entry
- Principal is a first-class auth role and redirects to /principal/dashboard after login

Bootstrap admin behavior:
- There is a default super admin fallback login
- It is intended as a platform-owner access fallback before full production auth is finalized
- This should not be treated as the final enterprise auth design

Auth implementation notes:
- Do not break restoreSession, hydrateAuthSession, or protected route behavior
- Preserve session hydration timing and loading states
- Respect role-aware redirect logic

IMPORTANT CURRENT PRINCIPAL IMPLEMENTATION DETAIL

The product request described nested Firestore staff documents under:
- schools/{schoolId}/staff/{staffId}

However, the current app architecture still uses school-scoped root collections with a schoolId field for operational compatibility across the existing services. The principal role was therefore implemented in the current school-scoped collection model instead of rewriting the whole data layer.

That means:
- Staff records remain in the existing staff collection
- Principal records are school-scoped via schoolId
- Principal uniqueness is enforced per school
- Auth custom claims carry role=principal and schoolId
- Public user profile role is principal
